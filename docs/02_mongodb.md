# MongoDB Data Model

## Design principle
In MongoDB, data is modeled around how it is read, not how it relates.
Every frequent query must be resolved with a single find() — no joins.

---

## Collection: `users`

**Purpose:** User management. One document per athlete.
The `email` field will be used for login once endpoints are secured.

```json
{
  "_id"          : "ObjectId",
  "name"         : "string",
  "email"        : "string (unique)",
  "password"     : "string (bcrypt hash)",
  "level"        : "beginner | intermediate | advanced | elite",
  "role"         : "ATHLETE | ADMIN",
  "weight_kg"    : "number",
  "height_cm"    : "number",
  "age"          : "number",
  "registered_at": "ISODate"
}
```

**Indexes:**
```
{ email: 1 }  — unique, login
```

---

## Collection: `plan_semana`

**Purpose:** Stores the plan generated by Gemini. Immutable after load.
One week = one document.

```json
{
  "_id"          : "ObjectId",
  "user_id"      : "ObjectId (ref: users)",
  "week_number"  : 10,
  "mesocycle"    : 3,
  "phase"        : "Realization / Specificity",
  "objective"    : "Weekly goal text",
  "start_date"   : "ISODate (Monday of the week)",
  "end_date"     : "ISODate (Sunday of the week)",
  "loaded_at"    : "ISODate (POST timestamp)",
  "athlete": {
    "name" : "string",
    "level": "beginner | intermediate | advanced | elite"
  },
  "training_days": [
    {
      "day"            : "Monday",
      "session_type"   : "string",
      "start_time"     : "07:00 | null",
      "session_day_id" : "ObjectId (ref: sesion_dia)",
      "blocks"         : [ "...see block structure below..." ]
    }
  ]
}
```

**Indexes:**
```
{ user_id: 1 }                   — plans for an athlete
{ week_number: 1 }               — find plan by week
{ start_date: 1 }                — find plan by date
{ mesocycle: 1, week_number: 1 } — list weeks of a mesocycle
```

---

## Collection: `sesion_dia`

**Purpose:** One document per training day. Contains the plan (copied from
plan_semana) and the actual result recorded by the athlete.
This is the primary collection consumed by Flutter.

```json
{
  "_id"          : "ObjectId",
  "user_id"      : "ObjectId (ref: users)",
  "week_plan_id" : "ObjectId (ref: plan_semana)",
  "week_number"  : 10,
  "mesocycle"    : 3,
  "day"          : "Monday",
  "date"         : "ISODate (real date of the day)",
  "session_type" : "string",
  "start_time"   : "07:00 | null",
  "status"       : "pending | completed | skipped",
  "post_notes"   : "string | null",
  "actual_rpe"   : "number | null",
  "completed_at" : "ISODate | null",
  "blocks"       : [
    {
      "...plan fields...",
      "result": "object | null"
    }
  ]
}
```

**Indexes:**
```
{ user_id: 1, date: 1 }                          — sessions for an athlete
{ date: 1 }                                       — today's session
{ week_plan_id: 1 }                               — 7 days of a week
{ week_number: 1, day: 1 }                        — specific day of a week
{ "blocks.exercises.name": 1, date: -1 }          — exercise history
{ status: 1, date: 1 }                            — pending/completed sessions
```

## Block structure (embedded in both collections)

All blocks share these base fields:
```json
{
  "order"                 : 1,
  "block_type"            : "string (see values below)",
  "target_rpe"            : 7.5,
  "heart_rate_zone"       : "Z3-Z4",
  "estimated_duration_min": 20
}
```

```json
{ "description": "free text" }
```
